# FAQs

## 1.为什么用 mtime+size 而不是直接比较哈希？

性能优化——SHA-256 需要读取全部文件内容，大文件 I/O 开销大；mtime+size 只需一次 `stat()` 系统调用即可获取，先用它做快速筛选，仅在二者之一变化时才计算哈希确认。

## 2.mtime 的局限性？

- mtime 精度通常为秒级（部分文件系统支持纳秒，但不保证），同一秒内的多次修改可能漏检
- `touch` 命令可人为修改 mtime，不能作为唯一依据
- 因此 mtime+size 只做初筛，最终依赖 SHA-256 内容哈希做确认判断

## 3.lettre 异步怎么和同步主循环配合？

主循环是同步的 `while` + `thread::sleep`，但 lettre 的 SMTP 发送是异步的。通过以下方式桥接：

1. 在 `cmd_run()` 中创建 `tokio::runtime::Builder::new_current_thread()` 轻量 runtime
2. 需要发送邮件时调用 `rt.block_on(mailer.send_xxx().await)` 阻塞当前线程等待异步完成
3. 邮件发送频率低（仅在启动/告警/退出时），阻塞不影响监控实时性

## 4.为什么选择 Rust？

- **内存安全**：所有权系统在编译期消除 use-after-free、double-free、数据竞争，无需 GC
- **零成本抽象**：高级语法（迭代器、闭包、模式匹配）编译后与手写 C 代码性能相当
- **类型系统**：`Option`/`Result` 强制处理空值和错误，`?` 运算符简化错误传播
- **Cargo 生态**：依赖管理、测试框架、文档生成一站式，clap/serde/sha2 等 crate 成熟稳定

## 5.基线文件（store.json）损坏了怎么办？

`store::load_baseline()` 在反序列化失败时会返回空的 `HashMap`，程序将其视为首次启动，重新对所有监控文件计算哈希建立基线，不会 panic 或丢失监控能力。

## 6.SMTP 发送失败了会影响监控吗？

不会。邮件发送失败只记录 ERROR 日志，不重试（等下个周期变化时再发），不丢失基线数据，不影响后续轮询。设计原则：**告警邮件是辅助通知手段，监控本身不能因邮件失败而中断**。

## 7.如何确保配置文件中的授权码安全？

- `fm init` 创建配置目录时自动设置 `700` 权限（仅 owner 可读写执行）
- 配置文件和数据文件设置 `600` 权限（仅 owner 可读写）
- 授权码明文存储是权衡——SMTP 协议要求发送方持有凭证，全盘加密会引入密钥管理复杂度
- 生产环境建议配合文件系统级加密（如 LUKS）或使用专用密钥管理服务

## 8.为什么不支持 Windows？

程序使用了 `std::os::unix::fs::PermissionsExt` 模块来设置文件权限（`0o600`/`0o700`），该模块仅在 Unix-like 系统可用。Windows 的权限模型完全不同（ACL），无法直接映射。
